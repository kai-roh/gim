# GIM (Geometry based Integrated Modeling) 해커톤 프로젝트 기획안

## 1. 프로젝트 개요

### 한 줄 정의

초고층 빌딩 기획 컨텍스트를 입력하면, 사용자가 선택한 거장 건축가 AI 패널이 토론을 통해 설계안을 확정하고, 수직 노드 그래프가 Rhino 3D 모델로 실시간 생성되며, 평가 모델의 피드백이 다시 토론장으로 돌아오는 순환 설계 시스템

### 핵심 질문

초고층 빌딩 설계는 수십 개 층의 프로그램 배분, 구조/코어 계획, 조망/바람/일조 등 환경 요소, 저층부 도시 접점까지 동시에 고려해야 하는 복합 의사결정이다. 단일 관점으로는 최적해에 도달할 수 없다. 만약 서로 다른 건축 철학을 가진 거장들이 하나의 프로젝트를 두고 실시간으로 토론하고, 그 결과가 즉시 3D로 구현되고, 구조/환경/법규 평가가 자동으로 피드백된다면, 초고층 설계의 초기 의사결정 품질은 근본적으로 달라진다.

### GIM vs BIM

|  | BIM | GIM |
|---|---|---|
| 단계 | 실시설계/시공 | 아이디어/기획설계 |
| 데이터 | 자재, 구조, MEP | 층별 프로그램, 공간 관계, 수직 동선 |
| 정밀도 | mm 단위 | 매스 레벨 (층 단위 그리드) |
| 목적 | 시공 문서화 | 설계 의사결정 지원 |
| AI 활용 | 간섭 체크, 물량 산출 | 배치 추론, 건축가 토론, 피드백 루프 |

윈도우 간 데이터 플로우:

- W1 → W2: 토론 합의 결과가 수직 노드/엣지로 자동 변환
- W2 → W3: 노드 그래프 변경 시 Rhino MCP를 통해 3D 매스 자동 재생성
- W3 → W2: Rhino에서 매스 수동 변경 시 노드 그래프 역반영
- W3 → W4: 생성된 3D 모델에 대해 평가 모델 자동 작동
- W4 → W1: 평가 피드백이 건축가 토론장에 자동 전달, 추가 논의 트리거
- Chat → W1: 사용자 의견이 건축가 토론장에 새로운 의제로 주입
- W2 노드 직접 편집 → W3 실시간 반영 (양방향 동기화)

## 3. 건축가 토론 시스템 (Architect Forum)

### 3.1 개요

건축가 토론장은 단순한 스타일 필터가 아니다. 서로 다른 건축 철학을 가진 AI 건축가들이 하나의 초고층 프로젝트를 두고 구조화된 토론을 수행하며, 합의 또는 대안을 도출하는 멀티에이전트 시스템이다. 사용자는 20인 클론 풀에서 프로젝트 성격에 맞는 2~5명을 직접 선택하여 토론 패널을 구성한다.

### 3.4 토론 프로토콜

```
[Phase 1: 발제]
사용자 컨텍스트 수신 → 각 건축가가 독립적으로 초기 제안 제시
    │
    ▼
[Phase 2: 교차 비평]
각 건축가가 다른 건축가의 제안에 대해 비평 및 대안 제시
    │
    ▼
[Phase 3: 수렴]
공통 합의점 도출 + 쟁점 정리
    │
    ▼
[Phase 4: 확정]
합의된 기획안 + 미합의 대안 목록 출력
    │
    ▼
[반복 트리거]
평가 피드백 수신 시 → Phase 2부터 재개
사용자 의견 수신 시 → Phase 1에 새 의제로 추가 후 Phase 2부터 재개
```

### 3.5 토론장 입출력 구조

입력 소스 3가지:

| 소스 | 시점 | 예시 |
|---|---|---|
| 사용자 컨텍스트 | 최초 입력 또는 추가 의견 | "저층부에 공공 보이드를 크게 넣고 싶어" |
| 평가 모델 피드백 | 3D 모델 생성 후 자동 | "50층 이상 풍하중 초과, 코어 면적 15% 부족" |
| 사용자 중간 개입 | 언제든지 | "Foster 의견이 마음에 들어, 그 방향으로" |

출력:

| 출력 | 형식 | 수신처 |
|---|---|---|
| 확정 기획안 | 구조화된 JSON | 수직 노드 그래프 생성기 |
| 토론 로그 | Markdown | W1 (사용자 열람) |
| 미합의 대안 목록 | JSON | SOT 저장 (추후 재참조) |

## 4. 수직 빌딩 노드 스키마 설계

### 4.1 개요

초고층 빌딩의 노드 그래프는 수평 건물과 근본적으로 다르다. 지하부터 옥상까지 수직으로 길게 늘어진 층별 구조이며, 각 층에 수평으로 여러 노드가 배치된다. 빌딩 자체의 형상처럼 그래프도 수직으로 길게 구축된다.

```
[옥상] ──── 헬리패드 ─── 전망대 ─── 기계실
  │
[고층 50~60F] ── 스카이라운지 ─── 호텔객실 ─── 코어
  │
[고층 40~49F] ── 호텔객실 ─── 코어 ─── 서비스
  │
[중간기계층 39F] ── 기계실 ─── 피난안전구역
  │
[중층 25~38F] ── 오피스 ─── 코어 ─── 회의실
  │
[스카이로비 24F] ── 스카이로비 ─── 라운지 ─── 셔틀EV
  │
[중층 15~23F] ── 오피스 ─── 코어 ─── 회의실
  │
[저층 6~14F] ── 프리미엄오피스 ─── 코어 ─── 지원시설
  │
[저층 1~5F] ── 로비 ─── 리테일 ─── 문화시설 ─── 공공보이드
  │
[지하 B1~B2] ── 주차 ─── 기계실 ─── 저장공간
```

### 4.2 노드 스키마 (초고층 특화)

```yaml
floor_node:
  # 식별
  id: string              # "F25_office_zone_A"
  name: string            # "25층 오피스 존 A"
  floor_level: integer    # 25
  floor_zone: enum        # basement, podium, low_rise, mid_rise, high_rise, sky_lobby, mechanical, crown, rooftop
  function: enum          # 아래 분류 참조

  # 물리적 제약
  constraints:
    area: number          # sqm (해당 층 내 점유 면적)
    height: number        # 층고 (m)
    position: string      # core_adjacent, perimeter, corner, center
    orientation: string   # N, S, E, W, NE, SE, SW, NW, all

  # 추상 속성 (0.0 ~ 1.0)
  abstract:
    publicity: number     # 공공성 (1F 로비 → 높음, 임원층 → 낮음)
    view_premium: number  # 조망 가치
    structural_load: number  # 구조 하중 중요도
    vertical_flow: number    # 수직 동선 의존도
    flexibility: number      # 용도 가변성
    prestige: number         # 위상/상징성

  # 비정형 태그
  tags: string[]          # ["랜드마크", "도시접점", "바람정원"]

  # 기하학 참조
  geometry_ref: string    # Rhino object GUID (nullable)
```

### 4.3 엣지 타입 (초고층 특화)

| 관계 | 의미 | 속성 |
|---|---|---|
| `STACKED_ON` | 수직 적층 (층간 관계) | structural_continuity: boolean |
| `ADJACENT_TO` | 같은 층 내 수평 인접 | wall_type: open / solid / glazed |
| `VERTICAL_CONNECT` | 수직 동선 연결 | via: elevator / stair / escalator / void |
| `SERVED_BY` | 기능적 종속 | 코어/기계층 → 오피스/호텔 |
| `VISUAL_CONNECT` | 시각적 연결 (아트리움/보이드) | degree: 0.0~1.0, spans_floors: integer |
| `ZONE_BOUNDARY` | 수직 존 경계 | transition_type: hard / gradient / sky_lobby |
| `FACES` | 외부 조건 대면 | target: view / road / adjacent_building / park |
| `STRUCTURAL_TRANSFER` | 구조 전이 (트랜스퍼 층) | system: outrigger / belt_truss / mega_column |

| 카테고리 | function enum 값 | 적용 존 |
|---|---|---|
| CORE | elevator_core, stair_core, service_shaft, mep_shaft | 전층 관통 |
| OFFICE | open_office, enclosed_office, co_work, executive_floor | 중/저층부 |
| HOTEL | guest_room, suite, hotel_lobby, banquet | 고층부 |
| RETAIL | retail_unit, anchor_tenant, food_hall | 저층부/포디움 |
| PUBLIC | lobby, plaza, public_void, gallery, observation | 저층부/옥상 |
| SOCIAL | sky_lobby, sky_garden, lounge, restaurant, fitness | 중간/고층부 |
| MECHANICAL | mechanical_floor, refuge_floor, cooling_tower | 중간기계층/옥상 |
| PARKING | underground_parking, loading_dock, car_lift | 지하부 |
| STRUCTURAL | transfer_floor, outrigger_level, mega_column_node | 구조 전이층 |

### 4.5 수직 노드 그래프의 차별점

| 일반 건축 그래프 모델 | GIM 수직 노드 그래프 |
|---|---|
| 단일 층 내 수평 관계 중심 | 지하~옥상 수직 스택 + 층별 수평 관계 |
| 공간 기능 분류만 존재 | 수직 존(zone) 개념: podium/low/mid/high/crown |
| 동선은 수평 이동 중심 | 수직 동선(EV, 스카이로비)이 1차 동선 |
| 구조가 그래프에 부재 | 구조 전이층/아웃리거가 노드로 명시 |
| 환경 요소 미반영 | 층별 풍하중/조망/일조가 속성으로 포함 |

### 4.6 3-그래프 구조 (Building-GAN 적용)

Autodesk AI Lab의 Building-GAN(ICCV 2021) 연구에서 제안된 3-그래프 건물 표현 체계를 GIM의 초고층 수직 노드 시스템에 적용한다. Building-GAN은 프로그램 그래프(버블 다이어그램)를 입력으로 3D 복셀 건물을 자동 생성하는 GNN 기반 모델이며, GIM 시스템의 "토론 합의 → 노드 그래프 자동 생성" 파이프라인과 구조적으로 대응된다.

**3-그래프 매핑:**

```
Building-GAN                          GIM
─────────────                         ───
Global Graph (FAR, 용도 비율)     ←→   Project SOT (대지 제약, 용적률, 프로그램 비율)
Local Graph (버블 다이어그램)      ←→   건축가 토론 합의 출력 (프로그램 인접 관계)
Voxel Graph (3D 공간 격자)        ←→   수직 노드 그래프 (층별 프로그램 배치)
Cross-Modal Attention             ←→   토론 합의 → 노드 그래프 자동 변환 브릿지
```

**프로그램 그래프 (건축가 토론 출력):**

건축가 토론의 합의 결과를 "어떤 용도가 어떤 용도와 인접/분리해야 하는가"라는 프로그램 인접 관계 그래프로 정형화한다.

```
[프로그램 그래프 예시]

  로비 ──(positive)── 리테일
   │                    │
(positive)          (positive)
   │                    │
  코어 ──(positive)── 공공보이드
   │
(negative: 분리)
   │
  기계실
```

- positive 엣지: 인접해야 하는 프로그램 쌍 (오피스↔코어, 로비↔리테일)
- negative 같은존 엣지: 같은 수직 존 내에서 분리해야 하는 쌍 (기계실↔로비)
- negative 다른존 엣지: 다른 수직 존 간 분리 제약 (지하주차↔호텔객실)

**층 위치 인코딩 (Positional Encoding):**

Building-GAN의 사인파 위치 인코딩(max 20층)을 초고층에 맞게 확장한다. 각 노드의 floor_level을 사인/코사인 함수로 인코딩하여, GNN 메시지 패싱 시 수직적 위치 관계를 학습 가능한 벡터로 표현한다.

```
PE(floor, 2i)   = sin(floor / 10000^(2i/d))
PE(floor, 2i+1) = cos(floor / 10000^(2i/d))
max_len: 150 (100층+ 초고층 대응)
```

**GIM 적용 시 핵심 변경점:**

| Building-GAN 원본 | GIM 적용 |
|---|---|
| 프로그램 타입 6개 | function enum 30+개 (초고층 특화) |
| 복셀 격자 50×40×40 (중저층) | 수직 중심 100+×20×20 (초고층 종횡비) |
| 층 위치 인코딩 max 20층 | max 150층 확장 |
| 수평 인접만 고려 | 수직 연속성 제약 추가 (코어/샤프트 정렬) |
| 단일 스케일 평가 | 3-스케일 평가: 층→존→빌딩 |

### 4.7 실시간 노드 편집

W2(수직 노드 그래프 윈도우)에서 노드를 직접 편집할 수 있으며, 변경이 실시간으로 W3(3D 모델)에 반영된다.

지원 편집 작업:
- 노드 추가/삭제: 특정 층에 새 프로그램 추가 또는 제거
- 노드 이동: 프로그램의 층 변경 (예: 스카이로비를 24F에서 30F로)
- 속성 변경: 면적, 층고, 추상 속성 값 실시간 조정
- 엣지 추가/삭제: 공간 관계 연결 변경
- 존 경계 변경: podium 끝나는 층수 변경 등

편집 → 반영 플로우:
```
W2 노드 편집 이벤트
    │
    ▼
노드 그래프 업데이트 (NetworkX)
    │
    ├─► W3 Rhino MCP: 변경된 노드/엣지만 delta 전송 → 3D 재생성
    │
    └─► W4 평가 모델: 변경 사항에 대한 즉시 재평가
            │
            ▼
        피드백 → W1 건축가 토론장
```

## 5. 평가 모델 (Evaluation Engine)

### 5.1 개요

3D 모델이 생성될 때마다 자동으로 작동하며, 정량적 평가 결과를 건축가 토론장에 피드백한다. 평가가 핵심인 이유는 건축가 토론이 감성적 논의에 머무르지 않고 정량적 근거 위에서 수렴하도록 강제하기 때문이다.

### 5.2 평가 차원

| 차원 | 평가 항목 | 평가 방식 |
|---|---|---|
| 구조 안정성 | 코어 비율, 슬래브 스팬, 종횡비, 풍하중 개략 검토 | 룰 기반 + LLM 추론 |
| 환경 성능 | 조망률, 일조 시뮬레이션(개략), 바람 영향, 에너지 등급 추정 | 파라미터 기반 |
| 법규 적합도 | 용적률, 건폐율, 높이 제한, 일조권, 피난 규정 | 룰 기반 |
| 경제성 | 연면적 대비 전용률, 프리미엄 층 비율, 주차 대수 충족 | 산식 기반 |
| 도시 맥락 | 저층부 공공 기여도, 보행 접근성, 주변 스카이라인 조화 | LLM 추론 |
| 3D 형태 평가 | Rhino 모델 기반 매스 비례, 입면 일관성, 구조체 정합성 | 3D 분석 룰 + MCP |
| 그래프 연결성 | 프로그램 인접 관계 유지율, 수직 연속성 정합도 | 그래프 분석 (Building-GAN 메트릭) |

> **연결성 정확도 (Connectivity Accuracy)**: Building-GAN에서 도입된 정량 메트릭. 건축가 토론에서 합의한 프로그램 인접 관계(positive 엣지)가 최종 노드 그래프에서 실제로 유지되는 비율을 측정한다. 예: 토론에서 "스카이로비↔셔틀EV 인접" 합의 → 노드 그래프에서 실제 인접 여부 검증. 이 메트릭은 토론 합의의 공간적 실현도를 정량화하며, 피드백 루프에서 "합의는 했지만 공간적으로 구현되지 않은 항목"을 자동 감지하는 데 활용된다.

**접점 정의**

| 접점 | 관련자 | 내용 |
|---|---|---|
| 토론 합의 ↔ 노드 그래프 변환 | 노현섭 + 이준원 | 토론 결과 JSON이 수직 노드/엣지로 정확히 변환되는지 |
| 클론 프로필 ↔ 토론 엔진 | 노현섭 (내부) | 20종 클론의 철학/성향이 토론에서 실제로 차별화되는지 |
| 노드 스키마 ↔ Rhino MCP | 이준원/조현호 + 백건호/심채윤 | 층별 노드 JSON이 Rhino 매스 생성 커맨드로 정확히 변환되는지 |
| 노드 편집 ↔ 3D 동기화 | 노현섭(UI) + 백건호/심채윤(MCP) | W2 편집 이벤트가 Rhino에 delta로 실시간 전달되는지 |
| 평가 모델 ↔ 피드백 루프 | 이준원/조현호(평가) + 노현섭(토론장) | 평가 JSON이 토론장에 자동 주입되어 다음 라운드를 트리거하는지 |
| 3D 평가 ↔ 종합 평가 | 백건호/심채윤(3D) + 이준원/조현호(종합) | Rhino 기반 3D 형태 평가가 종합 평가 점수에 통합되는지 |
| 사용자 개입 ↔ 토론장 | 노현섭 + 전원 | Chat 입력이 토론의 새 의제로 올바르게 주입되는지 |

첫 2시간에 데이터 인터페이스(JSON/YAML 스키마)를 전원이 합의하고, 이후 각자 모듈을 독립 개발하면서 접점에서 수시로 검증한다.

### 9.4 데모 클라이맥스

데모에서 가장 강력한 순간은 세 가지다:

1. 사용자가 20인 풀에서 건축가를 선택하고, 선택된 패널이 각자의 철학으로 토론을 시작하는 장면
2. 평가 피드백이 돌아와서 토론이 자동으로 재개되고, 수정된 3D가 즉시 업데이트되는 순환 루프
3. 사용자가 W2에서 노드를 직접 편집하면 Rhino 3D가 실시간으로 변하는 장면

데모에서 보여줘야 할 것:
- 20인 풀에서 건축가 패널 선택
- 자연어로 초고층 빌딩 컨텍스트 입력
- W1에서 선택된 건축가 패널 토론 실시간 진행
- W2에 수직 노드 그래프 자동 생성
- W3에 3D 매스 생성 (수직 스택)
- W4 평가 점수 표시 + 이슈 하이라이트
- 피드백 → 재토론 → 3D 수정 (루프 1회 이상)
- 사용자가 의견 제시 → 토론장 반영
- W2에서 노드 직접 편집 → W3 실시간 반영

데모에서 생략 가능한 것:
- 완벽한 환경 시뮬레이션 (개략 파라미터로 대체)
- 모든 법규 항목 체크 (핵심 3~4개만)
- 정밀한 구조 해석 (룰 기반 개략 검토로 대체)
- 20인 클론 전부의 토론 품질 (데모용 3~4명의 완성도에 집중)

## 10. 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 노드 그래프 저장 | NetworkX (해커톤) / Neo4j (프로덕션) | 빠른 프로토타이핑 우선 |
| LLM | Claude API (function calling) | 구조화된 JSON 출력 + 긴 컨텍스트 + 멀티에이전트 토론 |
| 토론 엔진 | Claude multi-turn + system prompt per clone | 클론별 독립 시스템 프롬프트로 성격 분리 |
| 평가 모델 | Python 룰 엔진 + Claude 추론 | 정량 룰 + 정성적 판단 혼합 |
| 3D 평가 | Rhino/GH 스크립트 + Python | 매스 비례, 구조 정합성, 풍하중 개략 분석 |
| 모델링 연결 | RhinoMCP | POC 검증 완료 |
| 프론트엔드 | Next.js | 4-윈도우 레이아웃 + 실시간 업데이트 |
| 그래프 시각화 | D3.js (수직 레이아웃 커스텀) | 수직 노드 그래프 특화 시각화 |
| 3D 웹 뷰어 | Three.js | 브라우저 기반 매스 확인 |
| 실시간 동기화 | WebSocket | 노드 편집 → 3D 반영 실시간 |
| 데이터 포맷 | JSON, YAML, CSV, Markdown | 구간별 최적 포맷 분리 |

**Phase 1: 해커톤 (24시간)**

- 초고층 복합타워 수직 노드 스키마
- 20인 건축가 클론 풀 (데모용 3~4명 완성도 집중, 나머지 기본 프로필)
- 패널 선택 → 멀티에이전트 토론 → 노드 그래프 → Rhino 3D 기본 파이프라인
- 평가 모델 기본 룰 (구조/법규/경제성) + 3D 형태 평가
- 피드백 루프 1회 이상 작동 시연
- 사용자 의견 → 토론장 주입 기능
- 4-윈도우 프로토타입 UI

**Phase 2: 해커톤 이후 1개월**

- 20인 전원 클론 품질 균등화
- 평가 모델 고도화 (환경 시뮬레이션, 에너지 성능 추정)
- 노드 실시간 편집 ↔ 3D 양방향 동기화 안정화
- 한국 초고층 건축법규 스킬 정밀화 (피난, 구조, 일조권)
- 토론 로그 기반 설계 의사결정 히스토리 DB 구축
- 웹 기반 독립 뷰어 (Rhino 없이 브라우저에서 확인 가능)

**Phase 3: 3개월**

- 건축사무소가 자사 포트폴리오로 커스텀 클론 생성 → 풀에 등록
- 분야별 전문가 에이전트 추가 (구조 엔지니어, MEP, 시공 관리)
- 클라이언트 에이전트 (예산/일정 제약 대변)
- 프로젝트 간 노드 그래프 학습 (유사 초고층 프로젝트 패턴 추천)
- 교육용 버전 (건축학과 초고층 설계 스튜디오)
- API 공개 (외부 모델링 도구, BIM 소프트웨어 연동)
